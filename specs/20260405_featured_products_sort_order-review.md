## HIGH

- [ ] **`items` 配列の最大件数制限が未定義** — セキュリティ / UX
  - バリデーションに `items` の上限がない。管理者アカウント侵害時のdefense-in-depthとして `items.length <= 50` 等の制約を追加すべき

- [ ] **認証方式の不一致: 仕様書と既存実装で異なるパターン** — DB・API / UX
  - 仕様は「NextAuth セッション + `role === "admin"`、未認証401/権限不足403」だが、既存APIは `getAdminSession()` で一律 `401` を返す
  - 既存の `getAdminSession()` パターンに合わせるよう仕様を修正すべき

- [ ] **`createProduct` / `updateProduct` での `featured_order` 自動設定ロジックの配置が曖昧** — DB・API / 要件完全性
  - 関数内部で自動計算するのか、route.ts ハンドラ側で計算して渡すのかが不明確
  - 特に `updateProduct` での featured 変更検知（0→1, 1→0）の実装方針を明記すべき

- [ ] **モバイルでのタッチセンサー設定が未定義** — UX
  - @dnd-kit の `TouchSensor` の `activationConstraint`（距離/遅延の閾値）が仕様にない
  - スクロール操作との競合防止のため、セクション 3.3.4 に設定値を明記すべき

## 判定

**結果: 不合格**

不合格理由:
- CRITICAL 1件（マイグレーション番号衝突）、HIGH 5件が残存
- 主な問題:
  1. マイグレーションファイル番号が既存と衝突し、本番適用時にエラーとなる
  2. APIバリデーションの方針（全件送信必須 vs 部分更新）が未定義で実装者が判断できない
  3. `featured_order` 自動設定ロジックの配置場所が曖昧で手戻りリスクが高い
  4. 認証パターンが既存コードと不一致

### 人間が判断すべき項目

1. **全件送信必須 vs 部分更新許容**: UIは全件送信する設計のため、全件必須（過不足で400エラー）を推奨
2. **Featured商品の上限数**: 並び替えUIのUX観点から20件上限を推奨

### 良い点

- スコープが「含む/含まない」で明確に整理されている
- バリデーションルールが6項目にわたり具体的でエラーメッセージまで定義されている
- 影響範囲テーブルで変更なし箇所も明記されている
- ライブラリ選定に比較表と不採用理由がある
- データ整合性シナリオ（同時編集、Featured解除中保存、欠番）が事前検討されている
- テスト計画が4レベル（ユニット/統合/E2E/手動）で具体的
- アクセシビリティ要件が仕様段階で考慮されている
