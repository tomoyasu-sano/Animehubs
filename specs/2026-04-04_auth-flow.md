# 認証フロー仕様書

## 概要

AnimeHubs の認証フロー全体を定義する。一般ユーザー・管理者の両方のログイン/ログアウト/セッション管理をカバーし、複数 Google アカウント利用時の UX 問題を解決する。

## 背景・経緯

管理者が2つの Google アカウント（`anytimes.snao.swe@gmail.com` と `anytimes.sano@gmail.com`）を併用していた。前者を管理者リストから削除後、Google OAuth が前回のアカウントを自動選択（`prompt=none`）し、管理者ログインできなくなった。暫定対応として `prompt: "select_account"` を常時設定したが、毎回アカウント選択画面が表示される UX 劣化が発生。

## 確定事項

| 項目 | 決定 |
|------|------|
| ログインプロバイダー | Google OAuth のみ（将来追加の可能性あり） |
| セッション戦略 | JWT（D1 + DrizzleAdapter 互換性のため — ADR-001 参照） |
| セッション有効期間 | 30日 |
| 管理者判定 | `ADMIN_EMAILS` 定数によるコード管理（変更時は constants.ts 編集 + 再デプロイ） |
| 並行セッション | 制限しない（JWT 戦略のため複数デバイスからの同時ログインを許可） |
| 「ログイン状態を保持」チェックボックス | 不要 |
| アカウント切り替え機能 | 不要 |

## 要件

### REQ-1: セッション有効時は自動ログイン

JWT が有効（30日以内）であれば、Google OAuth フローを経由せずにそのままアクセスできること。アカウント選択画面は表示しない。

### REQ-2: 初回ログイン時はアカウント選択

セッションがない状態でログインボタンを押した場合、Google アカウント選択画面を表示する。

### REQ-3: ログアウト後の再ログインでアカウント選択

ログアウト操作後、次のログイン時には Google アカウント選択画面を表示する。これにより、別のアカウントでログインし直せる。

### REQ-4: セッション期限切れ後はアカウント選択

JWT の 30日期限が切れた場合、次回アクセス時にアカウント選択画面を表示する。NextAuth が期限切れ JWT を検出すると自動的に Cookie を無効化し、未認証状態として扱う。

### REQ-5: 管理者ロールの自動付与

`ADMIN_EMAILS` に含まれるメールアドレスでログインした場合、JWT の role に `admin` を自動付与する。判定は JWT callback で毎回実行する（デプロイ後即反映）。

> **注意**: JWT callback が呼ばれるのはトークン生成・リフレッシュ時。middleware は JWT をデコードして role を読むのみ。ただし NextAuth v5 の JWT 戦略ではリクエストごとに jwt callback が実行されるため、`ADMIN_EMAILS` の変更はデプロイ後の次回リクエストで即反映される。

### REQ-6: 管理者パスの保護

`/admin/*`（`/admin/login` を除く）へのアクセスは、middleware で `role === "admin"` を検証する。非管理者は `/admin/login?error=not_admin` にリダイレクトし、「管理者アカウントでログインしてください」のメッセージを表示する。

### REQ-7: 保護パスの認証要求

以下のパスは認証必須。未認証時は `/auth/login?callbackUrl=...` にリダイレクトし、ログイン後に元のページへ戻す。

- `/cart`
- `/checkout`（`/checkout/complete` を除く）
- `/favorites`
- `/orders`
- `/account`

**callbackUrl のバリデーション**: 同一オリジンの相対パスのみ許可する。外部 URL へのリダイレクトは禁止（NextAuth v5 がデフォルトで同一オリジン制限を適用）。

### REQ-8: タブ間セッション同期

`SessionProvider` に `refetchOnWindowFocus: true` を設定し、タブ切り替え時にセッション状態を再検証する。これにより、タブ A でログアウト後にタブ B に戻った際、セッション状態が更新される。

## 認証フロー図

### 一般ユーザー

```
[ユーザーがサイトにアクセス]
        |
   JWT Cookie あり?
    /        \
  Yes         No
   |           |
JWT有効?    公開ページ → そのまま表示
  /    \    保護ページ → /auth/login にリダイレクト
Yes    No              |
 |      |         [ログインボタン押下]
 |   Cookie削除        |
 |   → /auth/login  Google OAuth
 |                  prompt: "select_account"
 |                  (state + PKCE 自動適用)
 |                     |
 |              [アカウント選択]
 |                     |
 |              JWT 生成（role付与）
 |              Cookie 保存（HttpOnly, Secure, SameSite=Lax）
 |                     |
 +---------+-----------+
           |
    [ページ表示]
    （以降30日間は自動ログイン）
```

### 管理者

```
[/admin にアクセス]
        |
   middleware チェック
        |
   JWT あり & role=admin?
    /        \
  Yes         No
   |           |
[管理画面表示]  [/admin/login にリダイレクト]
               |
         [Google ログイン]
         prompt: "select_account"
               |
         ADMIN_EMAILS に含まれる?
          /        \
        Yes         No
         |           |
   role=admin    role=user
   → /admin     → /admin/login?error=not_admin
                  （「管理者アカウントでログインしてください」表示）
```

### ログアウト

```
[ログアウトボタン押下]
        |
   signOut() 実行
        |
   JWT Cookie 削除
        |
   一般ユーザー → トップページ（/[locale]）
   管理者      → /admin/login
        |
   [次回ログイン時]
   → Google アカウント選択画面を表示
```

## 技術設計

### Google OAuth prompt パラメータの制御

**現状の問題**: `prompt: "select_account"` を常時設定しているため、毎回アカウント選択画面が表示される。

**あるべき姿**: `prompt` パラメータを条件に応じて切り替える。

| 状態 | prompt 値 | 挙動 |
|------|-----------|------|
| 初回ログイン（セッションなし） | `select_account` | アカウント選択表示 |
| ログアウト直後の再ログイン | `select_account` | アカウント選択表示 |
| セッション期限切れ後の再ログイン | `select_account` | アカウント選択表示 |
| セッション有効中 | N/A | OAuth フローに入らない（JWT で自動認証） |

**結論**: JWT が有効な間は OAuth フローに入らないため、`prompt: "select_account"` を常時設定しても**実質的な UX 影響はない**。OAuth フローが発動するのは以下の場合のみ:

1. 初回ログイン（セッションなし）
2. ログアウト後の再ログイン（Cookie 削除済み）
3. JWT 期限切れ（30日経過）

いずれも「アカウント選択を表示すべきタイミング」であるため、**現在の `prompt: "select_account"` 常時設定で正しい**。

### OAuth セキュリティ

- **state パラメータ**: CSRF 防止のため必須（NextAuth v5 が自動適用）
- **PKCE**: 認可コード横取り攻撃防止（NextAuth v5 が自動適用）
- **redirect URI**: Google Cloud Console に登録済みの URI のみ許可

### JWT 構造

```typescript
interface JWTToken {
  id: string;       // ユーザーID
  email: string;    // メールアドレス（JWE により暗号化）
  role: "admin" | "user";  // isAdminEmail() で毎回計算
  iat: number;      // 発行時刻（NextAuth 自動管理）
  exp: number;      // 有効期限（NextAuth 自動管理、発行から30日）
}
```

### JWT セキュリティ要件

| 項目 | 要件 |
|------|------|
| 署名アルゴリズム | HS256（NextAuth v5 デフォルト） |
| 署名鍵 | `AUTH_SECRET` 環境変数（最低32文字） |
| 暗号化 | JWE（NextAuth v5 デフォルト、ペイロード内の email 等を保護） |
| 鍵の環境分離 | 本番・ステージングで異なる `AUTH_SECRET` を使用 |
| Cookie 属性 | `HttpOnly`, `Secure`（本番）, `SameSite=Lax` |
| trustHost | `true`（Cloudflare Workers 環境のため必要） |

### セッション構造

```typescript
interface Session {
  user: {
    id: string;
    email: string;
    name?: string;
    image?: string;
    role: "admin" | "user";
  };
  expires: string;  // ISO 8601
}
```

> `src/types/next-auth.d.ts` で `Session["user"]` と `JWT` の型拡張済み。`as any` キャストは除去済み。

### 認証チェックポイント一覧

| レイヤー | 対象 | チェック内容 |
|---------|------|-------------|
| middleware | `/admin/*` | JWT 存在 + `role === "admin"` |
| middleware | 保護パス | JWT 存在 |
| API route | `/api/admin/*` | `getAdminSession()` で role 検証 |
| API route | 一般ユーザー向け API | `auth()` でセッション検証 |
| Client UI | AdminLayoutInner | `useSession()` で role 検証 |
| Client UI | AuthButton | `useSession()` で表示切り替え |

> 全ての `/api/admin/*` ルートで `getAdminSession()` に統一済み。

### DB テーブル（参考）

| テーブル | 用途 | JWT戦略での利用 |
|---------|------|----------------|
| users | ユーザー情報 | OAuth 初回ログイン時に作成 |
| accounts | OAuth トークン保存 | Google token 保存 |
| sessions | DB セッション | **未使用**（JWT 戦略のため。将来 DROP 予定） |
| verification_tokens | メール認証 | **未使用**（Google OAuth のみ。将来 DROP 予定） |

> **`users.role` カラムについて**: DB の `role` カラム（default: `"user"`）と JWT callback の `isAdminEmail()` が併存している。権威ソースは `ADMIN_EMAILS` 定数のみ。DB の `role` は参照されていない。混乱防止のため、将来的に DB の role カラムを削除するか、JWT と同期する仕組みを入れるべき。

### CSRF 対策

- 認証関連の状態変更（ログイン、ログアウト）は NextAuth の CSRF トークンにより保護
- `SameSite=Lax` Cookie により GET 以外のクロスサイトリクエストを防止
- アプリケーション独自の状態変更 API（カート操作、注文作成等）は NextAuth セッション検証で保護

## ADR（Architecture Decision Records）

### ADR-001: D1 + DrizzleAdapter で JWT 戦略を採用

- **背景**: D1 の database session に互換性問題が確認された
- **決定**: JWT 戦略を採用し、sessions テーブルは未使用とする
- **結果**: Cookie ベースでセッション管理、DB 接続不要で middleware が軽量
- **再評価条件**: D1 が datetime / transaction の制約を解消した場合

### ADR-002: 管理者判定を ADMIN_EMAILS 定数で管理

- **背景**: 管理者が3名で変更頻度が低い
- **決定**: `ADMIN_EMAILS` 定数を権威ソースとし、JWT callback で毎回計算
- **結果**: デプロイ即反映、DB 更新不要。コード変更 + デプロイが必要
- **再評価条件**: 管理者の頻繁な追加・削除が必要になった場合（環境変数化または DB 管理に移行）

## まとめ

現在の実装は `prompt: "select_account"` 常時設定で**機能的に正しい**。JWT セッションが有効な間は OAuth フローに入らないため、ログイン済みユーザーに毎回アカウント選択が出ることはない。OAuth フローが発動するのはセッションがない時だけであり、そのタイミングではアカウント選択を出すのがベストプラクティスである。

**`prompt: "select_account"` 常時設定はベストプラクティスに合致。** 加えて以下を実装済み:

- REQ-6: 管理者ログイン失敗時のエラーメッセージ表示
- REQ-8: タブ間セッション同期（`refetchOnWindowFocus`）
- TD-2: API 認証パターンの統一（`getAdminSession()`）
- TD-3: NextAuth 型拡張による型安全性向上
- TD-5: `getAuthHandler()` の Promise キャッシュ

残存する技術的負債:
- **TD-1**: `users.role` カラムと JWT role の二重管理（DB マイグレーション要）
- **TD-4**: 未使用テーブルの DROP（DB マイグレーション要）
- **TD-6**: 一般ユーザーログインページの i18n 対応
